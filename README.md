# Commander Deck Knowledge Base

Entry point for a fresh LLM session. Read this first, then pull specific docs as needed.

---

## What this project is

A structured knowledge base for building, scoring, and maintaining a themed Commander deck collection under a self-imposed Bracket 3 framework.

**Player context:** Based in Denmark. Physical collection ordered by card type — MDFC lands go into the lands pile; any multi-type card with "creature" as one of the types goes into the creatures pile. Deck files tracked via Moxfield exports and a custom `deck_safe_collection.xlsx`.

---

## Core constraints

- **Max 3 Game Changers per deck.** Cross-reference `REF_Game_Changers_List.md` every time. Never assume.
- **No mass land denial. No extra turn chains.** Hard exclusions, not preferences.
- **No early two-card infinites without pod approval.** Assume disallowed unless confirmed.
- **Target:** Bracket 4-adjacent power, Bracket 3-compliant by the letter.

---

## How to orient yourself

Read in this order if you're new to the project:

1. **`REF_The_Conversion_Check.md`** — the 20-point scoring framework. Every deck decision routes through this.
2. **`REF_Bracket_3_House_Rules.md`** — the explicit self-imposed rules.
3. **`REF_Domain_Principles.md`** — hard-won lessons. Read before making recommendations.
4. **`REF_Reskin_Aliases.md`** — Universes Beyond alias mappings. Check before declaring a card unowned.
5. **`REF_Game_Changers_List.md`** — authoritative GC list. Not memory.
6. **`Deck_Index.md`** — all active decks, scores, status at a glance.

## Ground truth hierarchy

When docs conflict, trust in this order:

1. **`.txt` decklist** — the actual current state of a deck.
2. **`_Summary.md`** — commentary on the decklist.
3. **`Collection_Master_Status.md`** — cross-deck state.
4. **Memory / prior chat** — lowest trust. Always verify against files.

## Naming convention

File prefixes are committed, not optional. They carry category information that folders alone can't express when files are flattened for Claude Project upload.

| Prefix | Category | Changes |
|---|---|---|
| `REF_` | Reference — stable framework and domain docs | Rarely |
| `WF_` | Workflow — repeatable skills / procedures | Rarely |
| `TPL_` | Template — skeleton for generated docs | Rarely |
| *(none, suffix `_Summary.md`)* | Deck summary | Per-deck audit |
| *(none, suffix `.txt`)* | Decklist, Moxfield-format | Per-deck swap |
| *(none, e.g. `Deck_Index.md`)* | Top-level state dashboards | Often |

## Conventions

- **Markdown everywhere.** Tables, headers, `---` section breaks, no decorative formatting.
- **Dates in filenames are `YYYYMMDD`.** Decklist `.txt` files: `<deck-kebab>-YYYYMMDD.txt`.
- **Deck names:** kebab-case in filenames, title case in prose.
- **UB reskins:** name the reskin first, original in parentheses. "Aang's Shelter (Teferi's Protection)".
- **Summaries are arguments, not inventories.** The `.txt` is the inventory. The summary explains the deck.

## Maintenance cadence

- **Per session:** update the relevant deck's `_Summary.md` and `.txt` if the deck changed.
- **After a batch of changes:** update `Deck_Index.md` and `Collection_Master_Status.md`.
- **Monthly (or after set releases):** verify `REF_Game_Changers_List.md` against WotC's current list. Regenerate collection spreadsheet from a fresh Moxfield export.
- **When a new lesson is learned:** append to `REF_Domain_Principles.md`.
- **When a new UB reskin is identified:** append to `REF_Reskin_Aliases.md`.

For the repo ↔ Claude Project sync cadence, see `SYNC.md`.
